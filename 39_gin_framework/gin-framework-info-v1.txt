# 📦 Golang GIN FRAMEWORK
════════════════════════════════════════════════════════════════════════════════════
1. OVERVIEW

-----------------------------------------------------------------

Gin is a high-performance HTTP web framework built on top of net/http.

Key Features:

* Fast (uses httprouter internally)
* Minimalistic & expressive
* Built-in middleware support
* JSON validation & binding
* Route grouping
* Error handling utilities

-----------------------------------------------------------------

2. INSTALLATION

-----------------------------------------------------------------

Step 1: Initialize module

```go
go mod init myapp
```

Step 2: Install Gin

```go
go get -u github.com/gin-gonic/gin
```

-----------------------------------------------------------------

3. BASIC SERVER (HELLO WORLD)

-----------------------------------------------------------------

```go
package main

import "github.com/gin-gonic/gin"

func main() {
	r := gin.Default()

	r.GET("/", func(c *gin.Context) {
		c.JSON(200, gin.H{
			"message": "Hello World",
		})
	})

	r.Run(":8080")
}
```

Key Concepts:

* gin.Default() → router + logger + recovery middleware
* c *gin.Context → request + response wrapper

19. COMPLETE MINI EXAMPLE

-----------------------------------------------------------------

```go
package main

import (
	"net/http"
	"github.com/gin-gonic/gin"
)

type User struct {
	Name string `json:"name" binding:"required"`
}

func main() {
	r := gin.Default()

	r.POST("/user", func(c *gin.Context) {
		var u User

		if err := c.ShouldBindJSON(&u); err != nil {
			c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
			return
		}

		c.JSON(http.StatusOK, u)
	})

	r.Run(":8080")
}
```

# INTERVIEW QUESTIONS (MED → SENIOR)

Q1: Why use Gin over net/http?
A:

* Faster routing
* Built-in middleware
* Cleaner syntax
* Less boilerplate

-----------------------------------------------------------------

Q2: What is gin.Context?
A:

* Wraps request and response
* Provides helper methods for JSON, params, etc.

-----------------------------------------------------------------

Q3: Difference between ShouldBindJSON and BindJSON?
A:

* ShouldBindJSON → returns error
* BindJSON → auto writes 400 response

-----------------------------------------------------------------

Q4: How middleware works in Gin?
A:

* Chain of handlers
* Uses c.Next() to continue execution

-----------------------------------------------------------------

Q5: How to handle validation in Gin?
A:

* Uses struct tags with validator
* binding:"required"

-----------------------------------------------------------------

Q6: How to structure scalable Gin app?
A:

* Layered architecture
* Dependency injection
* Separate router, handlers, services

-----------------------------------------------------------------

Q7: How to handle panics?
A:

* Recovery middleware (default)

-----------------------------------------------------------------

Q8: How to implement authentication middleware?
A:

* Check token in header
* Abort if invalid
* Call c.Next() if valid

-----------------------------------------------------------------

Q9: Is Gin thread-safe?
A:

* Yes, per request context is isolated

-----------------------------------------------------------------

Q10: When NOT to use Gin?
A:

* Extremely minimal apps → net/http enough
* Need full control over HTTP stack

-----------------------------------------------------------------

Q11: Performance comparison?
A:

* Gin faster than many frameworks
* Close to raw net/http performance

-----------------------------------------------------------------

Q12: How to write custom middleware chain?
A:

* r.Use(m1, m2, m3)

-----------------------------------------------------------------

Q13: How to test Gin handlers?
A:

* Use httptest
* Create router and send request

-----------------------------------------------------------------

Q14: What are common pitfalls?
A:

* Not handling errors properly
* Overusing global state
* Ignoring context cancellation

3. CORE CONCEPTS & EXAMPLES
--------------------------------------------------------------------------------

A. Grouping Routes (Versioning)
   v1 := r.Group("/api/v1")
   {
       v1.POST("/login", loginEndpoint)
       v1.GET("/user/:id", getUserEndpoint)
   }

B. Path Parameters & Query Strings
   // GET /user/john?action=send
   r.GET("/user/:name", func(c *gin.Context) {
       name := c.Param("name")
       action := c.DefaultQuery("action", "none")
       c.String(200, "Hello %s, action is %s", name, action)
   })

C. Model Binding and Validation
   type Login struct {
       User     string `json:"user" binding:"required"`
       Password string `json:"password" binding:"required,min=8"`
   }

   r.POST("/login", func(c *gin.Context) {
       var json Login
       if err := c.ShouldBindJSON(&json); err != nil {
           c.JSON(400, gin.H{"error": err.Error()})
           return
       }
       c.JSON(200, gin.H{"status": "logged in"})
   })

4. MIDDLEWARE (THE "MED-TO-SENIOR" CRITICAL AREA)
--------------------------------------------------------------------------------
Middleware in Gin is just a function with the signature `func(c *gin.Context)`.

Example: Custom Auth Middleware
-------------------------
func AuthMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        token := c.GetHeader("Authorization")
        if token != "secret-api-key" {
            c.AbortWithStatusJSON(401, gin.H{"error": "Unauthorized"})
            return
        }
        c.Next() // Continue to the next handler
    }
}
-------------------------

5. INTERVIEW QUESTIONS & ANSWERS (MID-SENIOR LEVEL)
--------------------------------------------------------------------------------

Q1: What is the difference between gin.Default() and gin.New()?
A1: gin.Default() initializes the engine with Logger and Recovery middleware already 
    attached. gin.New() creates a "blank" engine with no middleware. Use gin.New() 
    if you want full control over your middleware stack (e.g., custom logging).

Q2: How does Gin handle concurrency? Is it thread-safe?
A2: Like most Go web frameworks, Gin handles each request in its own Goroutine. 
    However, the `gin.Context` object is NOT thread-safe. If you start a new 
    Goroutine inside a handler to do background work, you MUST NOT use the 
    original context directly. You should use `c.Copy()` to get a read-only 
    copy for the background routine.

Q3: Explain the significance of c.Abort() vs return in a middleware.
A3: `return` only stops the execution of the current middleware function. 
    It does NOT prevent subsequent handlers in the chain from running. 
    `c.Abort()` ensures that the rest of the handlers/middleware for this 
    request are not called.

Q4: How do you implement graceful shutdown in a Gin application?
A4: You use the standard `http.Server` instead of `r.Run()`. You listen for 
    OS signals (SIGINT/SIGTERM) and then call `server.Shutdown(ctx)` with a 
    timeout context to allow active connections to finish.

Q5: Why is 'ShouldBindJSON' often preferred over 'BindJSON'?
A5: `BindJSON` automatically sets the response status to 400 and writes a 
    header if validation fails, which takes control away from the developer. 
    `ShouldBindJSON` returns the error, allowing the developer to manage 
    the response and logging manually.

Q6: How would you handle a global custom error response format?
A6: Create a custom middleware that uses `c.Next()`. After `c.Next()` returns, 
    check `len(c.Errors)`. If errors exist, format them into your standard 
    JSON structure and send the response.

6. PERFORMANCE NOTES
--------------------------------------------------------------------------------
- Gin uses `HttpRouter`, which uses a Radix Tree for routing. This is memory 
  efficient and has O(n) complexity where n is the length of the path.
- Minimize allocations by reusing structs or using `sync.Pool` if you are 
  handling tens of thousands of requests per second.
