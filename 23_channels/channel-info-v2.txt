# 🚀 Go Lang CHANNELS
════════════════════════════════════════════════════════════════════════════════════

1. CHANNEL BASICS
==================
Channels are Go's built-in mechanism for communication and synchronization
between goroutines. They follow the principle: "Do not communicate by sharing
memory; instead, share memory by communicating."

- Unbuffered channel:  ch := make(chan T)          → synchronous
- Buffered channel:    ch := make(chan T, size)    → asynchronous up to buffer

Key Rules:
- Always initialize channels with make(). The zero value (nil) blocks forever.
- Send operation:    ch <- value
- Receive operation: value := <-ch
- Close operation:   close(ch)   → only the sender(s) should close

2. UNBUFFERED vs BUFFERED CHANNELS
==================================
Unbuffered (synchronous):
- Sender blocks until a receiver is ready.
- Guarantees instant hand-off.
- Ideal for strict synchronization and data ownership transfer.

Buffered (asynchronous):
- Sender can continue until the buffer fills up.
- Useful for decoupling producers and consumers.
- Helps handle bursts of work.

Experienced Tip:
Prefer unbuffered channels unless you have a concrete performance or
back-pressure requirement. Buffers can hide race conditions and make
debugging harder.

3. CODE EXAMPLES

Example 1: Simple Unbuffered Producer-Consumer
-----------------------------------------------
package main
import "fmt"

func main() {
    ch := make(chan int)   // unbuffered

    go func() {
        for i := 0; i < 5; i++ {
            ch <- i
            fmt.Println("sent", i)
        }
        close(ch)
    }()

    for v := range ch {    // receives until closed
        fmt.Println("received", v)
    }
}

Example 2: Buffered Channel with Select
----------------------------------------
ch := make(chan string, 3)

go func() {
    ch <- "A"
    ch <- "B"
    ch <- "C"
    ch <- "D"   // this will block until someone receives
}()

for i := 0; i < 4; i++ {
    select {
    case msg := <-ch:
        fmt.Println(msg)
    }
}

Example 3: Worker Pool Pattern (Common in Production)
------------------------------------------------------
const numWorkers = 4
jobs := make(chan int, 100)
results := make(chan int, 100)

for i := 0; i < numWorkers; i++ {
    go worker(i, jobs, results)
}

// Send jobs
for i := 0; i < 20; i++ {
    jobs <- i
}
close(jobs)

// Collect results
for i := 0; i < 20; i++ {
    fmt.Println(<-results)
}

func worker(id int, jobs <-chan int, results chan<- int) {
    for job := range jobs {
        // simulate work
        results <- job * 2
    }
}

Example 4: Fan-In Pattern with Context (Senior Level)
------------------------------------------------------
func fanIn(ctx context.Context, inputs ...<-chan int) <-chan int {
    out := make(chan int)
    for _, in := range inputs {
        go func(ch <-chan int) {
            for {
                select {
                case v, ok := <-ch:
                    if !ok {
                        return
                    }
                    out <- v
                case <-ctx.Done():
                    return
                }
            }
        }(in)
    }
    return out
}

4. ADVANCED CONCEPTS & BEST PRACTICES
=====================================
- Channel Direction:
    - Send-only:   chan<- T
    - Receive-only: <-chan T

- Closing Channels:
    - Only close a channel if you are the sole sender.
    - Never close from the receiver side.
    - For multiple senders, use a separate "done" channel or context.

- The "comma-ok" idiom:
    value, ok := <-ch
    if !ok { // channel was closed and drained }

- Nil channels in select:
    A nil channel is never ready → useful for dynamically enabling/disabling cases.

- Common Patterns:
    - Worker pools
    - Fan-out / Fan-in
    - Pipeline processing
    - Timeout using select + time.After
    - Error handling via error channels or struct {Value T; Err error}

Senior Tips:
- Always check for closed channels with range or comma-ok.
- Combine channels with context.Context for cancellation.
- Use the -race flag during testing.
- Avoid large buffers; they can mask design issues.
- Prefer "for range ch" when the channel will be closed.
