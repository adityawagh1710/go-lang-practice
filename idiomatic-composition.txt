[1] CORE PHILOSOPHY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  • In Go, the concept of sub-typing or sub-classing does NOT exist.
  • These OOP design patterns should be actively AVOIDED.
  • The core principle is: group types by BEHAVIOR, not by what they ARE.
  • "Don't group types by a common DNA, group them by a common behavior."
                                                        — Bill Kennedy, Ardan Labs

  • Composition in Go is not just embedding structs — it's a complete design
    paradigm about how you architect systems using interfaces.

  • The goal: loose coupling, high flexibility, and code that can grow without
    breaking existing parts.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[2] THE ANTI-PATTERN — GROUPING BY TYPE (OOP-Style)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  • In Java/C++, you group things by a shared ancestor:
      - Animal → Dog, Cat (via inheritance / extends)
      - Base class holds common fields/methods
      - You store []Animal and treat subtypes polymorphically

  • In Go, developers coming from Java often try to replicate this:
      - Create a base struct (e.g., Animal)
      - Embed it into Dog, Cat
      - Store []*Animal and try to treat them as a group

  • This is an ANTI-PATTERN in Go. The compiler won't stop you from
    embedding, but embedding is NOT inheritance, and this approach
    leads to confusion and broken designs.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[3] WHY IT FAILS IN GO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  • Embedding is NOT inheritance. When Dog embeds Animal:
      - Dog does NOT become an Animal
      - A *Dog cannot be assigned to a variable of type *Animal
      - The compiler treats them as completely separate types

  • You cannot store *Dog and *Cat in a []*Animal slice.
    This would be a compile-time error.

  • Embedding only promotes fields and methods to the outer type for
    convenience — it is purely a composition/reuse mechanism.

  • There is no virtual dispatch through embedded structs. You can't
    override the base Speak() method and have it called polymorphically.

  • This design creates tight coupling with no real polymorphism benefit.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[4] THE GO WAY — GROUPING BY BEHAVIOR (Interfaces)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  • Define an interface that captures the BEHAVIOR you care about.
  • Any type that implements those methods automatically satisfies the interface.
  • Store []Speaker, []Mover, []Storer — groups of behavior, not type ancestry.

  • Benefits:
      - Loose coupling: types don't know about the interface
      - Easy extension: add a new type, it works if it has the right methods
      - Testable: easy to mock/stub with fake types implementing the interface
      - Idiomatic Go: this is how the standard library is designed

  • Key shift in thinking:
      FROM: "Dog IS-AN Animal"
      TO:   "Dog CAN-SPEAK"  ← interface captures the capability

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[5] IMPLICIT INTERFACE SATISFACTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  • In Java you must explicitly declare: "class Dog implements Speaker"
  • In Go, NO declaration is needed. If Dog has a Speak() method, it
    automatically satisfies the Speaker interface.

  • This is called STRUCTURAL TYPING (duck typing at compile time).

  • The developer writing Dog does NOT need to know about the Speaker
    interface. The interface and the type can be defined in completely
    separate packages, and the compiler will match them up.

  • The compiler does STATIC analysis to verify the match — no runtime
    overhead, full type safety.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[6] DISCOVERING vs DESIGNING INTERFACES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  • "Don't design with interfaces, discover them." — Rob Pike

  • WRONG approach (interface-first):
      - You start by writing an interface
      - Then write types to implement it
      - Interface is often too broad, too coupled to first implementation

  • CORRECT approach (concrete-first):
      - Write your concrete types solving the real problem
      - Once you see repeating behavior patterns, extract the interface
      - Interface is minimal, precise, and meaningful

  • This is why the Go standard library has tiny interfaces:
      - io.Reader  → just Read(p []byte) (n int, err error)
      - io.Writer  → just Write(p []byte) (n int, err error)
      - io.Closer  → just Close() error
      - fmt.Stringer → just String() string

  • Small, focused interfaces are powerful and composable.
    Large interfaces (many methods) are rigid and hard to satisfy.

[7] JAVA vs GO — COMPARISON TABLE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Concept               Java                          Go
  ─────────────────────────────────────────────────────────────────────────────
  Grouping mechanism    Base class / abstract class   Interface
  Type relationship     IS-A (inheritance)            CAN-DO (behavior)
  Declaration           "implements InterfaceName"    Implicit / structural

[8] CODE EXAMPLES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

── [8.1] BAD — OOP-Style Grouping (Concrete Base Type) ─────────────────────────

// Anti-pattern: trying to use a concrete base struct for polymorphism
// This is how a Java developer might (incorrectly) write Go

package main

import "fmt"

// Base "type" — mimicking an abstract Animal class
type Animal struct {
    Name string
}

// Method on base type
func (a *Animal) Speak() {
    fmt.Println(a.Name, "makes a generic sound")
}

// Dog embeds Animal
type Dog struct {
    Animal
}

// Cat embeds Animal
type Cat struct {
    Animal
}

func main() {
    dog := Dog{Animal{Name: "Rex"}}
    cat := Cat{Animal{Name: "Whiskers"}}

    // ❌ This does NOT work — []*Animal cannot hold *Dog or *Cat
    // animals := []*Animal{&dog, &cat}  // COMPILE ERROR

    // You're forced to do this (defeats the whole purpose):
    dog.Speak() // calls Animal.Speak() — not Dog-specific behavior
    cat.Speak() // calls Animal.Speak() — same output, no polymorphism

    // Even if Dog overrides Speak(), it won't be called through *Animal
    // because there's no virtual dispatch on concrete types in Go
    _ = dog
    _ = cat
}

// OUTPUT:
// Rex makes a generic sound
// Whiskers makes a generic sound


── [8.2] GOOD — Go-Style Grouping (Interface / Behavior) ───────────────────────

// Idiomatic Go: group by behavior, not by what a type IS

package main

import "fmt"

// Interface capturing the behavior we care about
// Named after what it DOES, not what it IS
type Speaker interface {
    Speak()
}

// Dog is a standalone concrete type — no base struct needed
type Dog struct {
    Name string
}

// Dog satisfies Speaker implicitly by having Speak()
func (d Dog) Speak() {
    fmt.Printf("%s says: Woof!\n", d.Name)
}

// Cat is a standalone concrete type
type Cat struct {
    Name string
}

// Cat satisfies Speaker implicitly
func (c Cat) Speak() {
    fmt.Printf("%s says: Meow!\n", c.Name)
}

// Robot can also be a Speaker — it shares behavior, not ancestry
type Robot struct {
    Model string
}

func (r Robot) Speak() {
    fmt.Printf("Robot %s says: Beep boop.\n", r.Model)
}

func main() {
    // ✅ Group by behavior — all implement Speaker
    speakers := []Speaker{
        Dog{Name: "Rex"},
        Cat{Name: "Whiskers"},
        Robot{Model: "R2D2"},
    }

    for _, s := range speakers {
        s.Speak() // polymorphic dispatch through interface
    }
}

// OUTPUT:
// Rex says: Woof!
// Whiskers says: Meow!
// Robot R2D2 says: Beep boop.


── [8.3] Polymorphic Function Using Interface ───────────────────────────────────

// Functions should accept interfaces, not concrete types
// This makes them work with ANY current and FUTURE type that satisfies the interface

package main

import "fmt"

type Speaker interface {
    Speak()
}

// MakeNoise accepts any []Speaker — decoupled from concrete types
func MakeNoise(speakers []Speaker) {
    for _, s := range speakers {
        s.Speak()
    }
}

// NotifyAll accepts a single Speaker — also decoupled
func NotifyAll(s Speaker) {
    s.Speak()
}

type Dog struct{ Name string }
func (d Dog) Speak() { fmt.Printf("%s: Woof!\n", d.Name) }

type Cat struct{ Name string }
func (c Cat) Speak() { fmt.Printf("%s: Meow!\n", c.Name) }

func main() {
    group := []Speaker{
        Dog{Name: "Rex"},
        Cat{Name: "Luna"},
    }

    MakeNoise(group)
    // MakeNoise works for ANY type with Speak() — no code changes needed
    // to add a new animal, robot, or any other type in the future
}


── [8.4] Discovering the Interface — Concrete First ────────────────────────────

// Step 1: Write concrete types solving the real problem (no interface yet)

type EmailNotifier struct {
    SMTPHost string
}

func (e *EmailNotifier) Send(msg string) error {
    fmt.Printf("Sending email via %s: %s\n", e.SMTPHost, msg)
    return nil
}

type SMSNotifier struct {
    PhoneNumber string
}

func (s *SMSNotifier) Send(msg string) error {
    fmt.Printf("Sending SMS to %s: %s\n", s.PhoneNumber, msg)
    return nil
}

// Step 2: You notice both types have Send(msg string) error
//         Now you DISCOVER the interface

type Notifier interface {
    Send(msg string) error
}

// Step 3: Write functions using the discovered interface
func Alert(n Notifier, message string) {
    if err := n.Send(message); err != nil {
        fmt.Println("Alert failed:", err)
    }
}

// This is idiomatic Go — the interface was discovered from real concrete usage,
// not designed upfront in isolation.

[9] INTERVIEW Q&A — MEDIOR LEVEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q1: Does Go support inheritance like Java?                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│ No. Go has no extends keyword and no sub-typing. Embedding a struct         │
│ promotes its fields and methods, but the outer type is NOT the inner type.  │
│ You cannot assign a *Dog to a *Animal variable even if Dog embeds Animal.   │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q2: How do you achieve polymorphism in Go?                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│ Through interfaces. Any type that implements all methods of an interface    │
│ automatically satisfies it. Functions accepting interfaces can work with    │
│ any conforming type — this is Go's polymorphism mechanism.                  │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q3: What is the difference between embedding and implementing an interface? │
├─────────────────────────────────────────────────────────────────────────────┤
│ Embedding copies field/method names into the outer struct for convenience.  │
│ It's a code reuse mechanism. Implementing an interface means having the     │
│ required method signatures — it's about behavior compatibility, not         │
│ structure. Embedding CAN help satisfy an interface (if the embedded type    │
│ has the needed methods), but they are separate concepts.                    │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q4: Do you need to declare that a type implements an interface in Go?       │
├─────────────────────────────────────────────────────────────────────────────┤
│ No. Go uses structural (implicit) typing. If a type has all the methods     │
│ defined by an interface, it satisfies it automatically. The compiler        │
│ validates this statically — no runtime cost, full type safety.              │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q5: What is the preferred size of a Go interface and why?                   │
├─────────────────────────────────────────────────────────────────────────────┤
│ Prefer small interfaces — ideally 1 to 3 methods. Small interfaces are      │
│ easier to satisfy, easier to compose, and more reusable. io.Reader,         │
│ io.Writer, and fmt.Stringer are famous examples. Large interfaces create    │
│ tight coupling and are hard for new types to satisfy.                       │
└─────────────────────────────────────────────────────────────────────────────┘

[10] INTERVIEW Q&A — EXPERIENCED LEVEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q1: What does Bill Kennedy mean by "discover interfaces, don't design them"?│
├─────────────────────────────────────────────────────────────────────────────┤
│ It means you should write concrete types first to solve a real problem,     │
│ then observe what common behaviors emerge across types. Once you see the    │
│ pattern (e.g., multiple types all have Send(msg) error), you extract a      │
│ minimal interface to represent that behavior. Designing interfaces first    │
│ tends to produce overly broad abstractions tied to the first implementation.│
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q2: What is "interface pollution" and how does it occur?                    │
├─────────────────────────────────────────────────────────────────────────────┤
│ Interface pollution is when interfaces are created unnecessarily or too     │
│ early — before there's real evidence that multiple types will share the     │
│ behavior. It leads to bloated interfaces, forced abstraction, and code      │
│ that's harder to understand. Occurs when developers try to "future-proof"   │
│ by designing interface hierarchies before writing any concrete types.       │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q3: Since Go has implicit interfaces, how can you verify a type implements  │
│     one at compile time without using it in a function call?                │
├─────────────────────────────────────────────────────────────────────────────┤
│ Use a blank identifier type assertion:                                      │
│                                                                             │
│   var _ Speaker = (*Dog)(nil)                                               │
│                                                                             │
│ This line creates no runtime value but forces the compiler to check that    │
│ *Dog satisfies Speaker. If it doesn't, you get a compile-time error.        │
│ Common pattern in library code and test files.                              │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q4: How does Go's interface-based grouping affect testability compared to   │
│     Java's inheritance-based approach?                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│ Go's approach is far more testable. Since functions accept interfaces,      │
│ you can pass in a mock/stub type in tests without any mocking framework.    │
│ Just create a local struct in the test file with the required methods.      │
│ In Java, mocking a class (not an interface) requires reflection-based tools │
│ like Mockito. Go's design naturally encourages dependency injection through  │
│ interfaces.                                                                 │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q5: Can embedding help a type satisfy an interface? Give an example.        │
├─────────────────────────────────────────────────────────────────────────────┤
│ Yes. If an embedded type satisfies an interface, the outer type also        │
│ satisfies it because the methods are promoted.                              │
│                                                                             │
│   type Logger struct{}                                                      │
│   func (l Logger) Log(msg string) { fmt.Println(msg) }                     │
│                                                                             │
│   type Service struct { Logger }   // embeds Logger                        │
│                                                                             │
│   type Loggable interface { Log(string) }                                   │
│                                                                             │
│   var _ Loggable = Service{}  // ✅ compiles — Log is promoted from Logger  │
│                                                                             │
│ This is intentional reuse via embedding, not inheritance. Service is NOT    │
│ a Logger; it just borrows its behavior.                                     │

┌─────────────────────────────────────────────────────────────────────────────┐
│ Q6: What is the memory layout difference between an interface and a         │
│     concrete type in Go?                                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│ An interface value is a two-word struct internally:                         │
│   - Word 1: pointer to the type descriptor (itype / *itab)                 │
│   - Word 2: pointer to the concrete value (or the value itself if small)   │
│                                                                             │
│ This means storing a value in an interface may cause a heap allocation      │
│ if the value is larger than a pointer. Calling a method through an          │
│ interface involves an indirect call (through the itab's function pointer    │
│ table), which is slightly less performant than a direct method call.        │
│ This cost is usually negligible but matters in tight hot paths.             │
└─────────────────────────────────────────────────────────────────────────────┘
